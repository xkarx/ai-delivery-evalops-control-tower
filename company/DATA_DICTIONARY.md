# DailyCart synthetic company data dictionary

> **Synthetic demo data.** Every identity, account, conversation, metric, release, and incident generated by this pipeline is fictional. Live workflow execution must label its source mode separately.

## Identity and lineage rules

Identifiers are stable within and across scenarios because they are assigned by entity position, never by random value. Scenario selection changes evidence content and distribution, not the identifier namespace.

| Prefix | Entity | Canonical file | Important links |
|---|---|---|---|
| `PER` | Persona | `generated/customers/personas.json` | Customer `personaId` |
| `CUS` | Persistent synthetic customer | `generated/customers/customers.json` | Research, support, bug, and analytics `customerId` |
| `EVD` | Normalized evidence | `generated/research/evidence.json` | Feature `evidenceIds`, source-record `evidenceId` |
| `FEAT` | Candidate or historical feature | `generated/product/features.json` | Decisions, backlog, releases, incidents, eval cases |
| `DEC` | Human decision fixture | `generated/product/decisions.json` | `featureId` |
| `TKT` | Delivery backlog item | `generated/product/backlog.json` | `featureId`, `dependsOn` |
| `REL` | Historical release | `generated/operations/releases.json` | `featureIds`, `evidenceIds` |
| `INC` | Historical incident | `generated/operations/incidents.json` | `featureId`, `evidenceId`, `regressionCaseId` |
| `EVALCASE` | Versioned eval case | `generated/evals/eval-cases.jsonl` | Feature, evidence, ticket, customer, or incident IDs in `input` |
| `ACT` | Product event | `generated/analytics/product-events.csv` | Persistent `customerId`; optional `featureId` in properties |
| `EXT` | Synthetic source-system record | Source-specific JSON files | Usually links to normalized `EVD` evidence |

`generated/lineage.json` contains explicit graph edges. Each feature-to-evidence reference must have an `EVD … supports … FEAT` edge. Incidents link to their regression cases.

## Common fields

| Field | Meaning |
|---|---|
| `sourceMode` | Always `synthetic` for this corpus. This is not a claim about later workflow execution. |
| `occurredAt`, `createdAt`, and other `*At` fields | ISO-8601 UTC timestamps from a fixed synthetic clock. |
| `evidenceId` | Link from a raw/source-shaped record to its normalized evidence record. |
| `evidenceIds` | One or more normalized evidence records supporting an entity. |
| `customerId` | Persistent synthetic customer identity used across files and product events. |
| `theme` or evidence tag | A discoverable problem family; it is an input to analysis, not a recommendation. |

## Directories and records

### `strategy/`

- `company-overview.md`: fictional company, scenario, date window, and data-use warning.
- `product-strategy.md`: strategic intent and prioritization guardrails.
- `quarterly-goals.json`: exactly three Q2 goal fixtures with baselines, targets, and owners.

### `customers/`

- `personas.json`: five behavioral personas with goals and constraints.
- `customers.json`: 50 persistent accounts. Emails use the reserved `.invalid` domain and cannot receive mail.

### `research/`

- `interviews.json`: interview metadata and transcript paths.
- `interviews/*.md`: eight labeled transcript excerpts with uncertainty and counter-signals.
- `evidence.json`: normalized evidence that conforms to the shared `evidenceSchema`.
- `surveys.json`: 12 purchase-confidence pulse responses with persistent customer and evidence links.
- `feature-requests.json`: 20 stated requests with importance and source-reliability qualifiers.
- `slack-threads.json`: eight synthetic cross-functional discussions.

Evidence `kind` is one of `interview`, `support`, `analytics`, `survey`, `incident`, or `discussion`. Sentiment is `positive`, `neutral`, `negative`, or `mixed`. Tags include a theme plus source type; some records are deliberately tagged `noisy`, `conflict`, `ambiguous`, `underpowered`, or `inconclusive`.

### `support/`

- `support-tickets.json`: 30 support records, with open and resolved states.
- `bugs.json`: 15 defects with severity, reproducibility, and affected customer links.

These are customer-support and defect-source records, not delivery `TKT` entities.

### `analytics/`

- `event-schema.json`: eight event definitions and required properties.
- `product-events.csv`: persistent-customer event stream generated from 120 sessions.
- `baseline-funnel.csv`: ordered session → view → cart → checkout start → completion counts.
- `experiments.json`: three historical tests, including inconclusive and underpowered results.

CSV `properties` is a JSON object encoded as a CSV cell. Counts are generated observations for the scenario, not live PostHog measurements.

### `product/`

- `features.json`: candidate and historical features conforming to the shared `featureSchema`.
- `backlog.json`: 25 linked delivery tickets conforming to `ticketSchema`.
- `decisions.json`: five human-decision fixtures conforming to `decisionSchema`.
- `roadmap.md`: stable-ID inventory; ordering is explicitly not priority.

The `score` field in a candidate feature is a deterministic signal-volume summary. It must not be treated as a PM recommendation.

### `operations/`

- `releases.json`: five synthetic historical releases.
- `incidents.json`: three resolved incidents linked to evidence, historical features, and critical regression cases.
- `postmortems/*.md`: three labeled incident narratives with impact, root cause, and corrective actions.

### `evals/`

- `eval-cases.jsonl`: 36 cases conforming to `evalCaseSchema`, covering grounding, requirements, build, regression, safety, and trajectory.
- `rubric.md`: shared grading expectations and critical-gate guidance.

Each JSONL line is a complete case. `datasetVersion` is `company-v1.0.0`; changing data generation semantics requires a version change.

## Validation invariants

`npm run validate:data` fails when a required file is missing, a minimum is not met, a shared schema fails, an ID is duplicated, a cross-file reference is invalid, timestamps exceed the synthetic clock, a funnel widens, a lineage edge is missing, the manifest disagrees with disk, or noise/conflict evidence is absent.
